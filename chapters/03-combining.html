3  Combining Data

So far you have worked with single datasets. In practice, the data you need is almost always spread across multiple files or sources. This chapter introduces two fundamental operations for combining data: concatenation (stacking datasets with the same columns on top of each other) and merging (joining datasets that share a common identifier column). These operations correspond to adding rows and adding columns, respectively.

We will use baseball data to illustrate both operations. Baseball has a long tradition of meticulous statistical record-keeping, and its datasets provide a clean and intuitive context for learning these skills. So we are going to take a break from last week’s very happy data about death statistics to work with baseball data this week.

The Retrosheet game logs and Lahman Database are the two most famous baseball datasets. Much of the data you see on sites like Baseball-Reference is derived from these sources.

3.1 Setup

import numpy as np
import pandas as pd
import os

pd.options.display.max_columns = 200

3.2 Setting and Resetting the Index

Before combining data, you need to understand how pandas uses the index. The index is the row label — by default an integer starting at 0, but it can be set to any column.

.reset_index() (docs) turns an index back into a regular column. .set_index() (docs) turns a column into the index.

batting_df = pd.read_csv('Lahman/Batting.csv')

batting_df.tail()

Make a pivot table of annual runs by team (review from Chapter 2).

team_annual_hrs = pd.pivot_table(
    data=batting_df,
    index='yearID',
    columns='teamID',
    values='R',
    aggfunc='sum'
)

team_annual_hrs.shape
team_annual_hrs.loc[1998:].dropna(axis='columns')

Reset the index to turn “yearID” back into a column.

reset = team_annual_hrs.reset_index()
reset

Set it back.

reset.set_index('yearID', inplace=True)
reset
reset.loc[2021].dropna().sort_values(ascending=False)
TipWhen to set an index

Set a column as the index when you will frequently use it for lookups (like dates or unique identifiers). Reset the index when you need that column available for merging or groupby operations.

3.3 Concatenation

A concat lets you add rows on top of each other when the columns are the same or similar. Think of it as stacking two spreadsheets that have the same column headers.

3.3.1 Checking the data first

Let’s start by loading a single season of game log data.

gamelog_1921 = pd.read_csv('Retrosheet/GL1921.TXT')
gamelog_1921.head(2)

The data does not have column names — pandas assumes the first row contains headers. We need to tell read_csv there is no header row.

gamelog_1921 = pd.read_csv('Retrosheet/GL1921.TXT', header=None)
gamelog_1921.head(2)

Now the columns are numbered 0 to 160, but the Retrosheet documentation indexes columns starting from 1. To avoid future confusion, let’s fix this now.

gamelog_1921.columns = range(1, 162)
gamelog_1921.head(2)

What is the “shape” of this data? The .shape attribute returns the number of rows and columns as a tuple.

gamelog_1921.shape

The 1921 data has 1,229 rows (games) and 161 columns. Compare to 2021 — almost twice as many games because of league expansion, but still 161 columns.

gamelog_2021 = pd.read_csv('Retrosheet/GL2021.TXT', header=None)
gamelog_2021.columns = range(1, 162)
gamelog_2021.shape

The 2020 season was shortened by the COVID-19 pandemic.

gamelog_2020 = pd.read_csv('Retrosheet/GL2020.TXT', header=None)
gamelog_2020.shape

A quick concatenation of two seasons:

pd.concat([gamelog_1921, gamelog_2021]).shape

3.3.2 Looping through files

With 151 season files, loading them one by one is impractical. The os.listdir function takes a directory path and returns a list of filenames. We will use it on the “Retrosheet” folder.

sorted(os.listdir('Retrosheet/'))

First, verify that all files have the same number of columns — a critical check before concatenating.

for file in os.listdir('Retrosheet/'):
    if 'gl' in file.lower():
        _df = pd.read_csv('Retrosheet/' + file, header=None)
        if _df.shape[1] != 161:
            print(f"WARNING: {file} has {_df.shape[1]} columns")

If nothing prints, all 151 files have exactly 161 columns. Now load them all into a list.

gamelog_list = []

for file in sorted(os.listdir('Retrosheet/')):
    if file.startswith('GL'):
        _df = pd.read_csv('Retrosheet/' + file, header=None)
        _df.columns = range(1, 162)
        gamelog_list.append(_df)

Develop a practice of constantly verifying that your operations match your expectations. We had 151 files, so our list should have 151 objects.

len(gamelog_list)
WarningValidate before concatenating

Always verify that all DataFrames you plan to concatenate have the same number of columns. A single mismatched file will introduce NaN values across hundreds of thousands of rows.

3.3.3 Using pd.concat

Concatenate all seasons into a single DataFrame.

gamelogs_df = pd.concat(objs=gamelog_list)

gamelogs_df.shape

3.3.4 Managing indices after concatenation

One interesting (and frustrating) consequence of concatenation is that the indices from each parent DataFrame are preserved. This means index values like 0, 1, 2, … repeat for every season.

gamelogs_df.loc[13]

There are 151 rows named 13 — one from each season. This is usually a problem. You have two options.

Option 1: Use .reset_index(drop=True) after concatenating. The drop=True parameter discards the old index rather than turning it into a column.

gamelogs_df_reset = gamelogs_df.reset_index(drop=True)

gamelogs_df_reset.loc[2000]

Option 2: Use ignore_index=True inside pd.concat. This is the lazy (and preferred) option because it handles the problem at the point of concatenation.

gamelogs_df_ignore = pd.concat(
    objs=gamelog_list,
    ignore_index=True
)

gamelogs_df_ignore.loc[1000]

3.3.5 Filtering concatenated data

With all 151 seasons in a single DataFrame, you can do interesting cross-historical analyses. Let’s extract the year from the date column and find all Boston Red Sox vs. New York Yankees games.

gamelogs_df_reset['year'] = gamelogs_df_reset[1].astype(str).str.slice(0, 4)
# Boston vs. Yankees games
away_condition = gamelogs_df_reset[4].isin(['BOS', 'NYA'])
home_condition = gamelogs_df_reset[7].isin(['BOS', 'NYA'])

sox_yankees_df = gamelogs_df_reset[home_condition & away_condition]
sox_yankees_df
pd.pivot_table(
    data=sox_yankees_df,
    index=4,
    columns=7,
    values=[10, 11],
    aggfunc='mean'
)

3.4 Merging

A merge lets you add columns next to each other when the rows share a common identifier. This is more commonly known as a join. Pandas has both a merge function and a .join() method, but merge is more general and is what we will use.

3.4.1 Preparing for a merge

Load the People and Batting tables from the Lahman database.

people_df = pd.read_csv('Lahman/People.csv')
print(people_df.shape)
people_df.head(2)
batting_df = pd.read_csv('Lahman/Batting.csv').sort_values(['playerID', 'yearID'])
print(batting_df.shape)
batting_df.head(10)

3.4.2 Identifying join keys

What columns do these two DataFrames have in common? Set operations on the column names reveal the shared keys.

set(batting_df.columns) & set(people_df.columns)

The playerID column is present in both. But do they cover the same set of players?

print(f"Unique players in batting: {len(set(batting_df['playerID']))}")
print(f"Unique players in people: {len(set(people_df['playerID']))}")
print(f"Players in both: {len(set(batting_df['playerID']) & set(people_df['playerID']))}")

There are more people in the People table than in the Batting table because the People table includes pitchers, managers, and other non-batters.

3.4.3 Checking for duplicates

Are there repeated player IDs in either table? Understanding this is critical for predicting the shape of your merged result.

people_df['playerID'].value_counts()

Each player appears exactly once in People — it is a lookup table. But in Batting:

batting_df['playerID'].value_counts()

Players appear multiple times because the Batting table has one row per player-season (and even per player-team within a season for traded players). This means our merge will be a one-to-many join: each person row will match multiple batting rows.

3.4.4 Types of joins

Pandas supports four types of joins through the how parameter.

3.4.4.1 Left join

A left join preserves all rows in the left DataFrame. Rows from the right that do not match get NaN.

people_batting_left = pd.merge(
    left=people_df,
    right=batting_df,
    left_on='playerID',
    right_on='playerID',
    how='left'
)

print(people_df.shape, batting_df.shape, people_batting_left.shape)
people_batting_left.head(12)

3.4.4.2 Right join

A right join preserves all rows in the right DataFrame. Rows from the left that do not match get NaN.

people_batting_right = pd.merge(
    left=people_df,
    right=batting_df,
    left_on='playerID',
    right_on='playerID',
    how='right'
)

print(people_df.shape, batting_df.shape, people_batting_right.shape)
people_batting_right.head(9)

3.4.4.3 Outer join

An outer join preserves all rows from both DataFrames. Non-matching rows get NaN on the side that did not match.

people_batting_outer = pd.merge(
    left=people_df,
    right=batting_df,
    left_on='playerID',
    right_on='playerID',
    how='outer'
)

print(people_df.shape, batting_df.shape, people_batting_outer.shape)

3.4.4.4 Inner join

An inner join preserves only rows where the key exists in both DataFrames. Everything else is dropped.

people_batting_inner = pd.merge(
    left=people_df,
    right=batting_df,
    left_on='playerID',
    right_on='playerID',
    how='inner'
)

print(people_df.shape, batting_df.shape, people_batting_inner.shape)

3.4.5 Validating the merge

Always check the shape of your merged DataFrame. Ask yourself: how many rows and columns should the result have? If the number of rows is unexpectedly large, you may have a many-to-many merge — which is almost never what you intended.

    right=people_df,

NoteExercise 5: Three-way join

Which schools produce the highest-paid players? Load the “Salaries”, “Schools”, and “CollegePlaying” tables from the Lahman database. Develop a strategy for joining all three and answering the question.

Solution:

salaries_df = pd.read_csv('Lahman/Salaries.csv')
college_df = pd.read_csv('Lahman/CollegePlaying.csv')
schools_df = pd.read_csv('Lahman/Schools.csv')

# Step 1: Join salaries to college playing on playerID
sal_college = pd.merge(salaries_df, college_df, on='playerID', how='inner')

# Step 2: Join schools on schoolID
sal_college_school = pd.merge(sal_college, schools_df, on='schoolID', how='left')

# Step 3: Aggregate
sal_college_school.groupby('name_full').agg(

This is definitely biased towards schools that produce a small number of players with enormous contracts.

## Merging {#sec-merge}

A `merge` lets you add columns next to each other when the rows share a common identifier. This is more commonly known as a **join**. Pandas has both a [`merge`](https://pandas.pydata.org/docs/reference/api/pandas.DataFrame.merge.html) function and a [`.join()`](https://pandas.pydata.org/docs/reference/api/pandas.DataFrame.join.html) method, but `merge` is more general and is what we will use.

### Preparing for a merge

Load the People and Batting tables from the Lahman database.

```{python}
people_df = pd.read_csv('Lahman/People.csv')
print(people_df.shape)
people_df.head(2)
```

```{python}
batting_df = pd.read_csv('Lahman/Batting.csv').sort_values(['playerID', 'yearID'])
print(batting_df.shape)
batting_df.head(10)
```

What columns do these two DataFrames have in common? Set operations on the column names reveal the shared keys.

```{python}
set(batting_df.columns) & set(people_df.columns)
```

The `playerID` column is present in both. But do they cover the same set of players?

```{python}
print(f"Unique players in batting: {len(set(batting_df['playerID']))}")
print(f"Unique players in people: {len(set(people_df['playerID']))}")
print(f"Players in both: {len(set(batting_df['playerID']) & set(people_df['playerID']))}")
```

There are more people in the People table than in the Batting table because the People table includes pitchers, managers, and other non-batters.

### Checking for duplicates

Are there repeated player IDs in either table? Understanding this is critical for predicting the shape of your merged result.

```{python}
people_df['playerID'].value_counts()
```

Each player appears exactly once in People — it is a lookup table. But in Batting:

```{python}
batting_df['playerID'].value_counts()
```

### Types of joins

Pandas supports four types of joins through the `how` parameter.

#### Left join

A left join preserves all rows in the left DataFrame. Rows from the right that do not match get `NaN`.

```{python}
people_batting_left = pd.merge(
    left=people_df,
    right=batting_df,
    left_on='playerID',
    right_on='playerID',
    how='left'
)

print(people_df.shape, batting_df.shape, people_batting_left.shape)
people_batting_left.head(12)
```

#### Right join

A right join preserves all rows in the right DataFrame. Rows from the left that do not match get `NaN`.

```{python}
people_batting_right = pd.merge(
    left=people_df,
    right=batting_df,
    left_on='playerID',
    right_on='playerID',
    how='right'
)

print(people_df.shape, batting_df.shape, people_batting_right.shape)
people_batting_right.head(9)
```

#### Outer join

An outer join preserves all rows from both DataFrames. Non-matching rows get `NaN` on the side that did not match.

```{python}
people_batting_outer = pd.merge(
    left=people_df,
    right=batting_df,
    left_on='playerID',
    right_on='playerID',
    how='outer'
)

print(people_df.shape, batting_df.shape, people_batting_outer.shape)
```

#### Inner join

An inner join preserves only rows where the key exists in *both* DataFrames. Everything else is dropped.

```{python}
people_batting_inner = pd.merge(
    left=people_df,
    right=batting_df,
    left_on='playerID',
    right_on='playerID',
    how='inner'
)

print(people_df.shape, batting_df.shape, people_batting_inner.shape)
```

### Validating the merge

Always check the shape of your merged DataFrame. Ask yourself: how many rows and columns should the result have? If the number of rows is unexpectedly large, you may have a many-to-many merge — which is almost never what you intended.

## Exercise 5: Three-way join

Which schools produce the highest-paid players? Load the "Salaries", "Schools", and "CollegePlaying" tables from the Lahman database. Develop a strategy for joining all three and answering the question.

**Solution:**

```{python}
salaries_df = pd.read_csv('Lahman/Salaries.csv')

# Step 1: Join salaries to college playing on playerID

# Step 2: Join schools on schoolID

    {'salary': 'mean'}
).sort_values('salary', ascending=False).head(10)
```
